## Executive Summary

This extension solves the #1 pain point on Windows: PI's bash tool blocks the terminal. We implemented Claude Code's exact behavior:

- **Blocking by default** (normal behavior)
- **Ctrl+B** sends any running command to background
- **5-minute auto-background** for forgotten commands
- **`/tasks`** to manage background jobs

**Current State:** Code complete, needs testing and refinement

## What This Project Is

### The Problem
PI Coding Agent on Windows is painful because:
1. Bash tool blocks the terminal completely
2. Long commands (npm install, builds, dev servers) freeze the UI
3. No way to continue working while commands run
4. Windows has no tmux/screen equivalent
5. Users are trapped waiting for commands

### The Solution
A PI extension that:
1. **Overrides the default bash tool**
2. **Runs commands synchronously** (normal blocking)
3. **Shows TUI monitor** during execution with elapsed time
4. **Detects Ctrl+B** via keyboard input handling
5. **Migrates to PowerShell job** when triggered
6. **Returns immediately** with job ID
7. **Provides `/tasks`** command for management

### Key Differentiator
Unlike existing PI philosophy ("use tmux"), this actually solves the problem for Windows users who can't/won't use tmux.

## Technical Implementation Details

### 1. Ctrl+B Detection Mechanism

**The Challenge:** PI tool execution blocks. How do we capture keyboard input?

**The Solution:** PI's TUI system provides `ctx.ui.custom()` which creates a focused component that can receive keyboard input.

**Implementation:**
```typescript
// During tool execution
if (ctx.ui?.custom) {
  ctx.ui.custom((tui, theme, keybindings, done) => {
    return {
      handleInput(data: string): void {
        // Ctrl+B detection
        if (data === '\x02' || data === 'ctrl+b') {
          // Signal backgrounding
          done('background');
        }
        
        // Also support Esc/Ctrl+C for cancel
        if (keybindings.matches(data, 'interrupt')) {
          done('kill');
        }
      },
      
      render(width: number): string[] {
        // Show status to user
        return [
          `⚡ Running: ${command}`,
          `⏱  Elapsed: ${elapsed}`,
          `Press Ctrl+B to send to background`,
          `(Auto-background in ${countdown}s)`
        ];
      }
    };
  });
}
```

**Why This Works:**
- TUI component has focus during execution
- `handleInput()` receives raw keyboard data
- `\x02` is the ASCII code for Ctrl+B
- `done()` signals the promise to resolve
- Execution returns control to agent

**Alternative Approaches Considered:**
- Global keybinding hook: Doesn't work during tool execution
- Signal handling: Windows doesn't support signals well
- Separate thread: Node.js is single-threaded
- **Chosen approach: TUI component is the only viable method in PI's architecture**

### 2. Process Migration Strategy

**The Challenge:** How to move a running process to background?

**The Solution:** Kill-and-respawn pattern

**Step-by-Step:**
```typescript
1. User presses Ctrl+B
   ↓
2. Kill foreground process
   child.kill('SIGTERM')
   ↓
3. Capture any output we have
   const capturedOutput = outputSoFar
   ↓
4. Spawn PowerShell job with same command
   Start-Job -ScriptBlock { & command }
   ↓
5. Transfer captured output to job
   Write to job's output file
   ↓
6. Create BackgroundJob record
   Add to completedJobs registry
   ↓
7. Start monitoring the job
   Poll every 2 seconds
   ↓
8. Return success to agent
   With job ID and status
```

**Trade-offs:**
- ✅ Reliable and simple
- ✅ Works with any command
- ✅ Clean separation of concerns
- ❌ Loses 1-2 seconds (process restart)
- ❌ Command restarts from beginning
- **Acceptance:** For long-running commands, 1-2s restart is negligible

**Why Not Process Hibernation?**
- Windows doesn't support process migration
- Would require complex checkpoint/restore
- Not reliable for all process types

### 3. PowerShell Job Implementation

**The Challenge:** Need true background execution on Windows

**The Solution:** `Start-Job` cmdlet with script generation

**PowerShell Script Template:**
```powershell
# Generated script: %TEMP%\pi-background-bash\job-{id}-script.ps1

# Set environment variables
$env:NODE_ENV = "production"
$env:API_KEY = "secret"
Set-Location "C:\Users\Name\Project"

# Execute command with output capture
try {
    $output = & npm install 2>&1
    $output | Out-File -FilePath "output.txt" -Encoding UTF8 -Append
    $LASTEXITCODE | Out-File -FilePath "exitcode.txt" -Encoding UTF8
} catch {
    $_ | Out-File -FilePath "error.txt" -Encoding UTF8
    "1" | Out-File -FilePath "exitcode.txt" -Encoding UTF8
}
```

**Why PowerShell Jobs?**
- True background process (runs in separate process)
- Invisible (no console window)
- Full lifecycle management (Start/Stop/Get/Remove)
- Built into Windows (PowerShell 5.1+)
- Better than alternatives:
  - `cmd /c start` - shows window
  - `spawn({detached: true})` - loses tracking
  - `nohup` equivalent - not native on Windows

### 4. Temp File Strategy

**Storage Location:** `%TEMP%\pi-background-bash\`

**File Structure:**
```
%TEMP%\pi-background-bash\/
├── job-1743691234567-1-output.txt     # stdout + stderr
├── job-1743691234567-1-error.txt      # PowerShell errors
├── job-1743691234567-1-script.ps1     # Generated PS script
├── job-1743691234567-1-exitcode.txt   # Exit code
└── jobs-state.json                    # Optional persistence
```

**Why Temp Files?**
- ✅ Survive PI restart
- ✅ Can read on-demand (no memory pressure)
- ✅ User can inspect with standard tools
- ✅ Simple to implement
- ❌ Needs cleanup strategy
- ❌ File I/O overhead (acceptable)

**Cleanup Strategy (Not Yet Implemented):**
- Auto-delete completed jobs after 24 hours
- On extension shutdown, offer to cleanup
- Configurable retention policy

### 5. Auto-Background Logic

**The Challenge:** Users might forget to press Ctrl+B

**The Solution:** 5-minute automatic backgrounding

**Implementation:**
```typescript
// In TUI monitor component
const timer = setInterval(() => {
  elapsed = Math.floor((Date.now() - startTime) / 1000);
  
  // Auto-background after 5 minutes (300 seconds)
  if (elapsed > 300 && status === 'running') {
    status = 'backgrounded';
    monitorResolve('background');
    done('background');
  }
}, 1000);
```

**Why 5 Minutes?**
- Matches Claude Code behavior exactly
- Good balance between:
  - Not too short (avoids interrupting intentional waits)
  - Not too long (prevents indefinite blocking)
- 5 minutes is "long enough to be annoying"

**User Override:**
- Can press Ctrl+B anytime before 5min
- Can configure with environment variable
- Can set `sync: true` to force blocking

## Key Design Decisions

### Decision 1: Extension vs Core Modification
**Choice:** Build as PI extension
**Status:** ✅ Implemented

**Rationale:**
- PI philosophy: minimal core, extensible
- Can be shared/installed by others
- No fork maintenance
- Uses PI's extension APIs properly

**Implementation:**
```typescript
pi.registerTool({ name: "bash", ... })  // Overrides default by name
```

### Decision 2: Windows-First (PowerShell Jobs)
**Choice:** PowerShell Start-Job for background execution
**Status:** ✅ Implemented

**Rationale:**
- Windows has worst blocking experience
- PowerShell has excellent job support
- No tmux/screen equivalent on Windows
- Can extend to Linux/Mac later

**Trade-offs:**
- ✅ Best Windows experience
- ✅ Native PowerShell integration
- ❌ Windows-only (for now)
- ❌ Requires PowerShell 5.1+

**Future:** Cross-platform support using `nohup` on Unix

### Decision 3: Blocking by Default
**Choice:** Commands run synchronously unless Ctrl+B pressed
**Status:** ✅ Implemented

**Rationale:**
- Matches Claude Code exactly
- No surprise behavior changes
- User is always in control
- Opt-in to backgrounding

**Trade-offs:**
- ✅ Predictable
- ✅ No breaking changes
- ✅ Aligns with PI philosophy
- ❌ Requires TUI for Ctrl+B
- ❌ Headless mode loses feature

### Decision 4: Kill-and-Respawn Migration
**Choice:** Kill foreground process, spawn as PowerShell job
**Status:** ✅ Implemented

**Rationale:**
- Reliable and simple
- Works with any command type
- Clean separation
- Windows doesn't support true process migration

**Trade-offs:**
- ✅ 100% reliable
- ✅ Simple implementation
- ❌ 1-2s restart delay
- ❌ Loses a bit of progress
- **Acceptance:** Negligible for long commands

### Decision 5: TUI Monitor Component
**Choice:** Show execution status with `ctx.ui.custom()`
**Status:** ✅ Implemented

**Rationale:**
- Only way to capture keyboard in PI
- Provides visual feedback
- Can show real-time output
- User sees countdown

**Trade-offs:**
- ✅ User knows what's happening
- ✅ Can Ctrl+B anytime
- ✅ Visual confirmation
- ❌ Requires interactive mode
- ❌ Adds UI overhead
