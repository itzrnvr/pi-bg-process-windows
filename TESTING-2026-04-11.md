## Why a Standalone Test Harness

Testing pi extensions normally requires: edit → build → copy → restart pi → run command → check result. This cycle takes 1-2 minutes per iteration.

The standalone test harness (`src/test.ts`) tests core logic **without pi running** by extracting the key functions and testing them directly. This gives sub-second feedback.

## What Gets Tested

| Test | What it verifies |
|------|------------------|
| resolveShell | Finds Git Bash at `C:\Program Files\Git\bin\bash.exe` |
| killTree | SIGKILL kills process instantly; orphan cleanup runs |
| Short command | `echo hello` completes normally in <10s, returns output |
| Error exit code | `exit 42` → `isError: true`, exit code in output |
| Long command → background | `sleep 30` → backgrounded after timeout, gets PID + log path |
| BashResult shape | `user_bash` handler returns correct `{ output, exitCode, ... }` shape |
| AbortSignal | Cancelling signal kills running process |
| Memory bounded | Backgrounded process streams to log file, not RAM |

## Architecture

```
src/
  index.ts    — Extension code (loaded by pi)
  test.ts     — Standalone test harness (runs independently)
```

The test harness duplicates the core functions from `index.ts` (resolveShell, killTree, executeWithTimeout) because `index.ts` is an extension module that requires pi's ExtensionAPI. The test versions have the same logic but without pi dependencies.

**When changing core logic in index.ts, update the matching functions in test.ts too.**

## Key Findings from Testing

### taskkill hangs from Node/Bun

`taskkill /T /F /PID` (the documented way to kill process trees on Windows) **hangs** when called via `execSync` or `execFileSync` from Node.js or Bun, targeting a child process of the current process. Both `execSync` and `exec` timeout after their configured limit.

**Workaround:** Use `process.kill(pid, "SIGKILL")` (calls `TerminateProcess` — instant) for the parent, then spawn a detached PowerShell process to find and kill orphans via `Get-CimInstance`.

### BashResult vs AgentToolResult

The `user_bash` event handler must return `{ result: BashResult }` where BashResult has fields like `output`, `exitCode`, `cancelled`, `truncated`, etc. Returning an `AgentToolResult` (with `content: [{ type: "text", text }]`) causes `textContent.unshift is not a function` crash in pi's `recordBashResult()`.
